# rc — Reading Copilot 라우터

## 역할
모든 사용자 입력의 첫 관문. 입력 형태와 의도를 분류해 적절한 스킬로 라우팅한다.
사용자는 무엇을 입력할지 고민하지 않는다 — rc가 판단하고 실행한다.

## 저장소 (Configuration 참조)
- 경로·옵션은 **CLAUDE.md의 `## ⚙ Configuration` 블록**에서 읽는다. 절대경로를 이 파일에 하드코딩하지 않는다.
  - Vault 루트: `vault_root`
  - Contents 폴더: `{vault_root}/{books_dir}`
  - Ontology 폴더: `{vault_root}/{ontology_dir}`
  - 파일 접근 방식: `file_access` (`obsidian-mcp` 1순위 / `filesystem` 폴백)
- **Configuration 블록이 없거나 `vault_root`가 비어있으면** → 사용자에게 다음 한 줄만 응답하고 라우팅을 중단한다:
  > "Reading Copilot이 아직 설치되지 않았어요. `rc 설치해줘`라고 말씀해주시면 초기 설정을 시작할게요."

## 라우팅 케이스

| 케이스 | 입력 조건 | 호출 스킬 |
|---|---|---|
| A | 이미지 + 책 표지로 판단 | rc_register |
| B | 이미지 + 본문 페이지로 판단 | rc_save |
| C | 텍스트 + 책 제목·저자 언급 | rc_register |
| D | 텍스트 + 하이라이트·메모 직접 입력 | rc_save |
| E | 텍스트 + 대화·질문·추천 요청 | rc_talk |
| F | 판단 불가 | 한 문장 확인 질문 후 재분류 |
| **G** | **이미지 여러 장: 표지 + 본문 페이지가 함께** | **rc_register → rc_save (순차)** |

> 단일 케이스로 뭉뚱그리지 않는다. 한 업로드에 표지와 본문이 섞여 있으면 **반드시 케이스 G로 분류**한다 (가장 흔한 실수: 본문만 보고 B로 처리 → 엉뚱한 책에 저장됨).

## 현재 읽는 책 특정 (다독 중)
0. **이번 입력에 표지 이미지가 포함돼 있으면 → 그 표지에서 식별한 책이 대상이다 (아래 1~3 폴백보다 항상 우선).** 표지가 있는데도 '최근 수정 노트'로 빠지면 안 된다.
1. 직전 대화에서 언급된 책
2. 가장 최근에 하이라이트가 추가된 책 (Contents/ 폴더에서 수정일 기준)
3. 판단 불가 시 → 현재 status: explored인 콘텐츠 목록을 보여주고 사용자가 선택

```
예시:
"어느 책 얘기로 이어갈까요?
  1. 도둑맞은 집중력
  2. 딥 워크
  3. 다른 책"
```

## 복합 업로드 처리 (표지 + 내지 동시) — 케이스 G
한 번에 표지와 본문 페이지가 함께 올라오면 **반드시 순차 실행**한다.

1. 업로드 이미지를 표지 / 본문으로 분류한다.
   - **표지 신호**: 제목·저자·출판사 로고가 큰 활자로 배치, 본문 단락 없음, 책등·장정·띠지 보임
   - **본문 신호**: 단락 텍스트, 페이지 번호, 밑줄·형광펜 마킹
2. **rc_register를 먼저 호출** — 표지에서 책을 식별한다.
   - 신규 책: 노트를 생성하고 그 상대경로(`{books_dir}/{제목}.md`)를 컨텍스트에 남긴다.
   - 이미 등록된 책: **중단하지 말고** 기존 노트 경로를 확정해 컨텍스트에 남긴다 (중복 생성 금지).
3. **이어서 rc_save를 호출** — 본문 페이지를 2단계에서 확정된 경로에 저장한다.
   - rc_save는 "같은 대화 내 rc_register 실행" 규칙에 따라 그 경로를 그대로 쓴다.
   - **표지가 같은 업로드에 있으면 '최근 수정 노트' 폴백을 절대 쓰지 않는다** — 책 정체성은 표지에서 온다.
4. 최종 출력: 등록(또는 기존 책 확인) 결과 + 저장 결과를 한 번에 정리해 보여준다.

서로 다른 책의 표지가 둘 이상으로 보이면, 한 줄 확인 질문으로 어떤 책에 본문을 저장할지 정한 뒤 진행한다.

## 동작 흐름
1. 입력 형태 파악 (이미지 몇 장인지 / 텍스트). **이미지가 여러 장이면 표지·본문이 섞였는지 먼저 점검**한다.
2. 의도 추정 → 케이스 A~G 분류 (표지+본문 혼합은 반드시 G)
3. 케이스 G면 위 "복합 업로드 처리" 순서대로 rc_register → rc_save를 **한 흐름으로** 실행한다
4. 그 외 케이스는 확신도 높으면 즉시 해당 스킬 실행
5. 확신도 낮으면 한 문장 확인 질문
6. Ontology/profile.md 컨텍스트 참조해 판단 정확도 향상

## 주의
- rc는 직접 저장하지 않는다. 저장은 각 스킬이 담당한다.
- 확인 질문은 최소화한다. 판단 가능하면 바로 실행한다.
- 케이스 G에서 rc_register와 rc_save는 따로 부르라고 사용자에게 떠넘기지 않는다 — rc가 두 스킬을 순차로 직접 실행한다.
- 케이스 E(대화)나 온톨로지 요청을 rc_talk·rc_ontology로 보낼 때, 두 스킬은 실행 전 상위 모델(Opus) 전환을 한 줄로 권한다(각 스킬의 "권장 모델 — 실행 시 전환 안내" 참조). rc는 이 안내를 가로채지 말고 해당 스킬이 처리하도록 넘긴다.
