# Ontology 규칙집 (Methodology)
_LLM Wiki 버전의 표준 규칙. rc_ontology가 이 문서를 따른다._
_제정: 2026-06-25_

> 이 문서는 온톨로지의 **헌법**이다. 명명·연결·점수·갱신·고아 처리의 단일 기준.

---

## 1. 엔티티명 & 파일 표준화 규칙

### 1.1 엔티티 타입과 저장 위치
| 타입 | 정의 | 저장 위치 | 그래프 노드 |
|---|---|---|---|
| **Work** | 책·영화·다큐 = 작품 | `Books/{제목}.md` (노트 자체) | 노트 파일 |
| **Concept** | 책을 가로지르는 개념 | `Ontology/Concepts/{개념명}.md` | 개념 파일 |
| **Person** | 저자·인물 | `Ontology/people.md` (섹션) | 참조용(비노드) |
| **Claim** | 근거에 앵커링된 명제 | `Ontology/claims.md` (섹션) | 참조용(비노드) |

- **Work 노드는 새로 만들지 않는다.** Books/의 노트가 곧 Work 노드다(중복 노드 금지).
- **Concept만 개별 파일**로 둔다 → `[[개념명]]` 위키링크가 실제 노드로 해석되고, Obsidian 백링크가 양방향성을 자동 생성한다.
- Person·Claim은 그래프 폭증을 막기 위해 그룹 파일에 섹션으로 둔다(2차 참조).

### 1.2 명명 규칙 (링크 해석의 핵심)
1. **위키링크 타깃 = 파일명과 정확히 일치**해야 한다. 불일치 시 링크가 죽는다.
2. **Work**: 파일명 = 사용자가 정한 제목 그대로 (`[[테니스 이너 게임]]`, `[[Devil Wears Prada 2]]`, `[[소금. 산. 지방. 불]]`).
3. **Concept**: 한국어 명사구, 공백 허용, 특수문자 회피 (`몰입`, `정보 네트워크`, `자아1과 자아2`). 파일명 = 링크명.
4. **별칭(alias)**: 한 작품이 여러 이름으로 불리면 노트 프론트매터에 `aliases: ["원제", "다른 표기"]`를 두어 링크가 해석되게 한다.
5. 프론트매터의 모든 엔티티 참조는 반드시 `"[[...]]"` **따옴표+위키링크** 형식 (YAML 리스트가 Obsidian 링크로 인식되도록).

### 1.3 책 노트 `ai_*` 프론트매터 스키마 (표준)
```yaml
ai_entity_type: Work
ai_concepts:                 # 이 작품이 인스턴스화하는 Concept (→ Concepts/*.md)
  - "[[몰입]]"
ai_connected_works:          # 개념을 공유하는 다른 Work (→ Books/*.md)
  - "[[씻다르타]]"
ai_cross_themes:             # 연결의 '근거'가 된 브리지 개념 (ai_concepts의 부분집합, 반드시 존재하는 Concept 파일)
  - "[[자아1과 자아2]]"
ai_interest_score: 0.00      # 아래 3장 공식
ai_summary: "한 문장 요약"
ai_last_analyzed: 2026-06-25
```
- `ai_*`만 rc_ontology가 쓴다. LAYER 1·2(Human)는 불가침.
- 불문율 재정의: **불변 대상 = Human 하이라이트·인상·대화 발화.** `ai_*` 프론트매터는 원래부터 AI 파생 레인이므로 여기에 쓰는 것은 불문율 위반이 아니다.

## 2. 독자 관점 (profile.md)
LLM Wiki는 *세계*를 모델링하지 *독자*를 모델링하지 않는다는 약점이 있다. 이를 보완하기 위해 `Ontology/profile.md`를 **독자 레이어**로 유지한다.
- Concept/Work 노드에 **독자 관심 가중치**를 얹어 "무엇에 관심 있고 어디가 공백인가"를 복원한다.
- 추천·공백 탐지(PRD가 온톨로지에 기대한 1차 기능)는 이 파일이 담당한다.
- 자세한 구조는 `profile.md` 참조.

## 3. `ai_interest_score` 산출 공식
재현 가능해야 한다. 임의 감(感)으로 매기지 않는다.

```
score = min(1.0,  0.04·H  +  0.06·E  +  0.20·D)

  H = 하이라이트/메모 항목 수          (분량)
  E = 감정 마킹 수 (?!, !!, 💭, 메모 라인)  (몰입도)
  D = 서사 깊이   0 = 없음
                 1 = 짧은 메모 위주
                 2 = 긴 '한줄 인상' 또는 '나의 생각' 서사 존재
```
- 예) 넥서스 H≈25·E다수·D1 → 0.95(캡) · 청년파산 H0·D2 → 0.44 · Gattaca 짧은 인상 → 0.20
- 데이터가 없으면 0.10 고정(스텁). 아래 4·5장 참조.

## 4. 하이라이트 갱신 규칙 (재분석 트리거)
새 하이라이트가 들어올 때 온톨로지를 언제·어떻게 갱신하는가.

| 트리거 | 실행 범위 | 동작 |
|---|---|---|
| 하이라이트 5개 누적(5,10,15…) | 해당 Work | `ai_interest_score` 재계산 + `ai_concepts` 재추출 |
| 새 개념이 **2번째 Work**에서 등장 | Cross-book | 그 개념을 **Concept 파일로 승격**(§5) + 양쪽 `ai_connected_works` 갱신 |
| 완독 선언(status→explored) | Work + User | `ai_summary` 확정 + profile.md 관심 가중치 갱신 |
| 7일 무입력 | User | profile.md 시간축·공백 갱신 |
| 사용자 명시 요청 | 전체 | 3레이어 재분석 |

**갱신 원칙**
1. **증분(incremental)**: 기존 노드를 지우지 않고 병합·가중치 조정. 링크 무결성 우선.
2. **멱등(idempotent)**: 같은 하이라이트를 다시 분석해도 결과가 동일. (rc_save.py의 지문 중복제거와 정합)
3. `ai_last_analyzed` 타임스탬프를 매 갱신 시 갱신한다.
4. Human 데이터는 절대 수정하지 않는다(읽기만).

## 5. 고아 노드 금지 규칙 (Orphan Policy)
그래프 건강도를 위해 **연결되지 않는 노드는 생성하지 않는다.**

- **Concept 승격 조건**: 아래 중 하나를 만족해야 파일로 만든다.
  1. **2개 이상의 Work**에서 등장, 또는
  2. **2개 이상의 다른 Concept**과 연결됨.
- 위 조건을 못 채우는 개념(1개 Work에만 있고 다른 개념과도 안 엮임)은 **Concept 파일로 만들지 않고**, 해당 Work의 `ai_concepts`에 인라인 태그로만 남긴다.
  - 예) `맛의 4요소`는 [[소금. 산. 지방. 불]] 한 작품에만 있고 다른 개념과 안 엮여 → 파일 미생성(인라인 태그 유지).
- **스텁 정책**: 하이라이트·인상 데이터가 전무한 Work(듀얼 브레인·RAFA·참교육·파이 이야기 등)에는 **개념 노드를 생성하지 않는다.** `ai_interest_score: 0.10`, `ai_concepts: []`, `ai_summary`에 "데이터 부족" 명시. **환각 방지가 최우선.**
- 고아 탐지: rc_ontology는 주기적으로 Concepts/ 중 링크 차수 0인 파일을 찾아 사용자에게 보고(자동 삭제 금지).
