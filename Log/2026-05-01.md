지난 한 달 남짓, 막연한 컨셉 한 줄에서 시작해 v0.2 baseline까지 끌고 왔다. 큰 줄기는 이렇게 흘렀다.

> **컨셉 발화 → 서비스 정의서 v3.0 → PRD/기능명세서/디자인정의서 → 4문서 체계 정착 → v0.1 baseline → Doodle 디자인 시스템 풀 적용 → v0.2 → GitHub 공개 준비(README·CHANGELOG·Releases)**

그리고 그 사이사이에 자산 관리 자동화 워크플로우, 스킬 패키징, 폴더 체계 정정 같은 "본 프로젝트는 아니지만 일하는 방식 자체를 다듬는" 작업들이 끼어들었다.

## 1단계 — 컨셉을 문서로 옮기다

처음엔 _"흩어진 관심의 조각들을 모아 엮으면, 거기서 또 다른 나의 생각을 길러낼 수 있을까?"_ 이 한 문장만 가지고 있었다. 이걸 코드로 옮기기 전에 먼저 구조를 잡아야 한다고 봤다.

**서비스 정의서 v3.0**부터 그렸다. 단일 입력창 기반, 로컬 에이전틱 독서 앱(Mac 전용)으로 방향을 잡았고, 5개 에이전트 체계(`AG-OR` Orchestrator / `AG-BD` Book Discriminator / `AG-TE` Text / `AG-DG` Dialogue / `AG-IS` Insight Synthesis)를 설계했다. 각 에이전트에 모델을 다르게 매핑한 게 핵심이었다 — Haiku는 라우팅에, Sonnet은 본문 처리에, Opus는 인사이트 합성에. 토큰 효율과 응답 속도를 동시에 잡는 방향.

그 다음 **PRD**와 **기능명세서**를 차례로 작성했다. 처음엔 파일을 다른 폴더에 만들어 나중에 `app_build/`로 옮기는 등 자잘한 위치 정리도 같이 했다.

## 2단계 — 4문서 체계 정착

작업이 쌓이면서 문서가 4개로 굳어졌다.

|문서|역할|
|---|---|
|`service_def.md`|WHY · 개념|
|`PRD.md`|WHAT + HOW (통합 스펙)|
|`design_def.md`|VISUAL + UX|
|`aboutme.md`|메타/온톨로지(ID 매트릭스, 호환성 로그)|

이 체계가 자리잡으면서 **doc-coherence-check** 스킬을 따로 만들었다. 4문서 사이의 AG-ID, FR-ID, DS-ID, SCR-ID 무결성을 자동으로 검증하는 읽기 전용 도구. 이후 **doc-change-intent-detect**(어느 층위 문서부터 고칠지 분류)와 **doc-change**(실제 편집·전파) 도 같이 만들면서, 변경 요청이 들어오면 _진단 → 편집 → 검증_ 으로 흐르는 워크플로우를 만들었다.

작업하다 발견한 사소한 참조 오류도 그때그때 잡았다 — `DS-READING-STATUS`가 사라졌는데 PRD에 잔재로 남아있던 케이스, `aboutme §4.3` 앵커가 `§4.4`로 옮겨갔는데 §5.1 표가 업데이트 안 된 케이스 등. Patch 수준은 호환성 로그에 행 추가 없이 처리하는 룰도 정착시켰다.

## 3단계 — v0.1 baseline → v0.2 (Doodle 디자인 시스템)

v0.1까지는 4문서가 뼈대만 있었다. v0.2에서 본격적으로 **Doodle 디자인 시스템**을 풀 적용했고, design_def는 §1·§2·§5·§6를 처음으로 구체화하면서 Major 버전이 됐다. PRD도 FR-UI-13~15, FR-IS-16, DS-METADATA 3필드, AG-OR surface, §10.8 룰테이블이 들어가면서 Minor 상승. service_def는 Ambient Persistence 보정과 케이스 E 응답 귀속 정도만 들어간 Patch.

옵션 Y(그래프 렌더링은 react-native-svg 우선, FPS·hit-testing 한계 시 @shopify/react-native-skia 폴백)도 이때 결정했다.

이 과정에서 폴더 체계를 한 번 잘못 잡았다가 정정하기도 했다. `docs/v0.2/`를 만들었는데, "현재 작업 중인 버전은 latest와 중복이니 별도 폴더 두지 말자"는 결론으로 다시 지웠다. **룰을 명확히 박았다.**

```
docs/latest/   ← 항상 최신 작업본 (지금은 v0.2)
docs/v{X.Y}/   ← 직전 버전 baseline 스냅샷 (수정 금지)
```

v0.3 출시 시점이 돼야 비로소 `docs/v0.2/`가 생긴다. 이걸 `VERSION_RULES.md`와 `RELEASES.md`에 명시적으로 적어둬서 다음에 헷갈리지 않게 했다.

## 4단계 — Expo Go로 첫 빌드를 손에 쥐다

3개 산출물(PRD, 기능명세서, 디자인 정의서 등)을 인풋으로 Claude Code에서 코드를 짰고, **Expo Go로 앱을 처음 돌려봤다**. 초안치고는 나쁘지 않다고 느꼈다.

여기서부터 고민은 _"개선을 어떻게 운영할 것인가"_ 로 옮겨갔다:

1. **버전관리** — GitHub 필수, 폴더 복사 말고 git tag(`release-v0.1`, `release-v0.2`)로 식별
2. **스킬 폴더** — `app_build/` 밖에 `.claude/skills/`로 분리 (스킬은 코드 버전과 독립적으로 진화)
3. **개선 우선순위** — UI 먼저, 모델 최적화는 그 다음. 사용자 체감이 바뀌어야 모델 효과도 눈에 보임
4. **토큰 효율** — ♥︎(=PRD/기능명세서/디자인정의서/개발파일) 전체 재작성 말고 패치 형태로만 적용

UI 레이아웃과 지식 그래프 표시 이슈도 잡았다(coherence check PASS).

## 5단계 — GitHub 공개 준비

처음 세상에 내놓을 채비를 했다.

**README**는 두 톤을 오가며 다듬었다. 처음엔 약간 정돈된 톤으로 시도했다가 _"원래 표현과 어미, 말투를 최대한 유지하되, 논리정연하게 재구성"_ 으로 톤을 다시 잡았다. "박살난 집중력", "릴스를 넘기고 있다", "부스러기 관심", "벌크업" 같은 내 입말은 살리고, 문단 사이의 인과만 정돈했다. 헤드라인은 _"기억에 새겨지는 책읽기를 위한 AI 코파일럿"_ 으로.

**릴리스 노트 체계**도 표준에 맞췄다. README 하단에 박는 대신 3-layer 구조로:

- `CHANGELOG.md`(루트, Keep a Changelog 포맷, 단일 출처)
- GitHub Releases 탭(태그 + 빌드 첨부)
- README 본문 맨 아래엔 _"버전 이력은 CHANGELOG.md 참고"_ 한 줄만

첫 릴리스 타이틀은 `v0.1.0 — 첫 공개`로 정했고, Release notes 초안에는 What's Included / Known Limitations / What's Next / Getting Started 4섹션을 넣었다. _"첫 버전인데 완벽한 척하는 프로젝트보다, 한계를 알면서도 공개한 프로젝트에 더 별이 붙는다"_ 는 감각으로 Known Limitations를 솔직하게 비워뒀다.

## 회고

가장 잘한 결정은 **4문서 체계와 ID 무결성 검증을 일찍 자동화**한 것이다. 바이브코딩만으로 가다 보면 변경할 때마다 어디부터 손대야 할지 매번 헷갈렸을 텐데, 지금은 _"이 변경은 WHY 층위인가, WHAT인가, VISUAL인가"_ 로 분류만 하면 나머지는 워크플로우가 끌고 간다.

가장 가치 있던 자각은 _"♥︎ 전체 재작성 말고 패치 형태로 적용"_ 룰이다. 초반엔 변경요청 한 번에 4문서를 통째로 다시 쓰던 패턴이 굳을 뻔했는데, 패치 단위로 끊어내면서 토큰도, 내 머리도, 둘 다 살았다.

가장 헷갈렸던 건 폴더 체계였고 — `docs/v0.2/` 만들었다 지운 그 한 사건이 _"현재 작업본은 latest 단일, 이전만 버전 폴더"_ 룰을 굳혀줬다.

다음 단계는 결국 **직접 쓰면서 발견하는 불편함**이 동력이 될 것이다. 도그푸딩으로 부딪힌 지점부터 v0.3 변경 요청을 채워나가자.
